# 로켓스타트 — 실전 제품화 비판적 리뷰

> "많은 사람이 실제로 쓰는 제품"이 되려면 무엇이 부족한가.
> 요청: 코드에서 모자란 부분 + 제품화에 필요한 부분을 **비판적으로**.

## 0. 전제 (읽기 전 확인)

- 이 리뷰 시점에 리뷰어는 `dream0grow/rocketstart-web` 저장소와 라이브 프리뷰에
  **직접 접근하지 못했습니다**(세션 GitHub 권한이 `dream0grow/rocketstart`로 한정,
  프리뷰 URL은 403). 따라서 아래 "코드 결함"은 **동일 제품의 스타터 코드**를 근거로 한
  **강한 추정**이며, `⚠️확인` 표시가 붙은 항목은 실제 코드에서 확인이 필요합니다.
- "제품화 체크리스트"(2장 이후)는 저장소와 무관하게 이 제품(미루기 극복 · 뽀모도로 ·
  Next+Supabase · 한국 사용자)에 그대로 적용됩니다.

## 1. 코드에서 거의 확실히 모자란 부분 (⚠️확인 필요)

### 🔴 치명적 — 핵심 기능이 "가짜"일 위험
| # | 의심 지점 | 왜 치명적인가 | 확인/수정 |
|---|---|---|---|
| C-1 | **데이터가 실제로 저장되는가** | mock 데이터·localStorage로만 돌면 "새로고침/다른 기기에서 사라짐" → 제품이 아님 | 로켓 생성 → Supabase에 INSERT → 다시 조회되는지 |
| C-2 | **로그인이 실제로 붙었는가 + RLS가 진짜 막는가** | 인증 없이 열리거나 RLS 미검증이면 **남의 데이터가 보임**(치명적 유출) | 로그아웃 상태 접근 차단, 다른 계정 데이터 조회 시도로 RLS 검증 |
| C-3 | **타이머가 백그라운드 탭에서 정확한가** | 뽀모도로 앱의 생명. `setInterval`은 백그라운드 탭에서 **throttling되어 느려지거나 멈춤** → 25분이 30분처럼 감 | "시작 시각 기준(timestamp) 계산"으로 바꾸고, 필요시 Web Worker |

> **C-3은 특히 중요합니다.** 스타터의 `FocusTimer`는 `setInterval(1000)`으로 1초씩
> 빼는 방식이라, 사용자가 다른 탭·다른 앱을 보면 브라우저가 타이머를 늦춥니다.
> **집중 앱인데 집중하는 동안(=탭을 벗어난 동안) 시간이 틀립니다.** 반드시 고쳐야 합니다.

### 🟠 중요 — 자체 브리프가 요구했는데 빠졌을 것들
- **C-4 세션↔로켓 연결**: `focus_sessions.task_id`는 NOT NULL인데, 타이머에서 "어느 로켓을
  집중하는지" 고르지 않으면 저장 불가/무의미. ⚠️확인
- **C-5 하위 과업(Subtask) 입력**: 예상 뽀모도로는 스키마상 **subtask**에 있는데, 폼이
  **deliverable에 직접** 받으면 데이터 모델과 어긋남(이전 리뷰 P1-3). ⚠️확인
- **C-6 "90% 완료 확정" 버튼**: `is_90_done`/`confirmed_90_at`을 실제로 세팅하는 UI가 있는지. ⚠️확인
- **C-7 복구/성찰 저장**: 복구 모드 입력이 `reflection_logs`로 실제 저장되는지(스타터는 mock). ⚠️확인

### 🟡 숨은 버그 (스타터 코드에서 확인된 것)
- **C-8 타임존 버그** ⭐: `new Date().toISOString().slice(0,10)`은 **UTC 날짜**를 줍니다.
  한국(KST, +9)에서 **밤 9시 이후엔 날짜가 하루 밀립니다.** 로켓 시작일·D-day·"오늘 화면"이
  자정 근처에 틀어집니다. → 로컬 타임존 기준으로 날짜를 만들어야 함.
- **C-9 백색소음**: 볼륨 조절 없음(0.18 고정), 브라우저 자동재생 정책(사용자 제스처 필요),
  합성 노이즈 품질이 낮음 → 실사용 시 "소리가 별로다"로 이탈.
- **C-10 타이머 완료 피드백 없음**: 25분이 끝나도 소리·알림·뽀모도로 +1이 없음 → 언제 끝났는지 모름.

## 2. 제품화 — "많은 사람이 쓰려면" 비판적 체크리스트

여기부터가 요청의 핵심입니다. **MVP가 도는 것과, 남에게 내놓는 것은 다릅니다.**

### 🔴 P0 — 이게 없으면 "공개하면 안 되는" 수준

**2-1. 개인정보·법적 준비 (한국은 특히 강함)**
- 이메일·구글 로그인 = **개인정보 수집** → **개인정보처리방침**과 **이용약관**이 *법적으로 필수*.
- **회원 탈퇴 + 데이터 완전 삭제** 기능(개인정보보호법상 파기 의무).
- **데이터 내보내기**(사용자 자산: 로켓·성찰 기록).
- Supabase **데이터 저장 리전**(가급적 서울/도쿄)과 저장 항목을 방침에 명시.
- 성찰 메모엔 사용자의 **감정·심리 정보**가 담김 → 민감하게 취급, 최소 수집·암호화 고려.
> 미루기·심리 관련 앱은 "내 약점을 적는 곳"입니다. **신뢰가 곧 제품**입니다.

**2-2. 보안**
- **RLS 실제 검증**(위 C-2). 뷰(`rocket_task_metrics`)도 사용자별로 막히는지.
- `SUPABASE_SERVICE_ROLE_KEY`가 **클라이언트 번들에 절대 안 들어가는지**(서버 전용).
- 입력 검증·서버측 권한 확인(클라이언트만 믿지 않기).
- 비밀번호 재설정·이메일 인증·세션 만료 처리.

**2-3. 핵심 습관 루프 = 알림 (이 제품의 존재 이유)**
- 미루는 사람을 위한 앱인데 **넛지(알림)가 없으면 사람은 안 돌아옵니다.** 이건 부가기능이 아니라 *본질*.
- 웹 푸시: VAPID 키 + 서비스워커 등록 + 권한 요청 UX + **백엔드 스케줄러**(reminder_events 발송)까지 있어야 실제로 울림.
- ⚠️ **iOS 사파리 제약**: 홈 화면에 추가한 PWA만 푸시 가능, 제약 많음 → iOS 사용자 넛지 전략 별도 필요.
- 스타터 `sw.js`엔 푸시 핸들러만 있고 **등록·구독·발송이 없음** → 지금은 안 울림.

### 🟠 P1 — 초기 사용자 유지(리텐션)에 직결

**2-4. 온보딩 & 첫 성공 경험**
- 타깃이 "미루는 사람" → **가입 직후 막히면 바로 이탈**. 빈 화면 대신
  "예시 로켓 1개 미리 채워두기", "3분 만에 첫 로켓" 플로우.
- 첫 세션에서 **작은 성공(15분 최소 시작 완료)**을 반드시 경험하게 설계.

**2-5. 측정 없이는 개선 없음 — 분석 & 모니터링**
- 브리프의 KPI(착수율·복구율·예측정확도·재방문율)를 **실제로 측정**할 이벤트 로깅
  (Plausible/PostHog/GA 등, 개인정보 최소).
- **에러 모니터링**(Sentry) — 사용자는 버그를 신고하지 않고 그냥 떠납니다.
- 사용자 피드백 창구(간단한 문의/버그 버튼).

**2-6. 신뢰성 & 오프라인**
- PWA **아이콘**(192/512) 없으면 설치 배너·홈 추가 안 됨(스타터 `icons:[]`).
- 네트워크 끊겨도 타이머·현재 화면은 동작하고, 복구 시 동기화.
- 낙관적 업데이트 + 실패 시 롤백(저장 실패 무시하면 데이터 유실).

### 🟡 P2 — 규모가 커지면 필요

- **접근성**(a11y): 라벨·대비·키보드·스크린리더. 색만으로 상태 구분 금지.
- **성능**: 로켓·세션이 수백 개일 때 리스트·리포트 쿼리(인덱스, 페이지네이션).
- **운영/배포**: 스테이징 환경, DB 마이그레이션 관리, **자동 백업**, CI(빌드·타입·테스트·린트 게이트).
- **비용/비즈니스 모델**: 사용자가 늘면 Supabase·호스팅·푸시 비용 발생. 무료/유료 경계,
  또는 지속 가능성 계획.
- **랜딩 페이지 & SEO**: 앱 밖에서 "이게 뭐하는 앱인지" 3초 설명 + 가입 유도.
- **i18n**: 지금은 한국어 전용 — 확장 시 문자열 분리.

## 3. 제품 관점의 냉정한 질문 3개

1. **"미루는 사람"이 이 앱을 여는 습관 자체를 어떻게 만들 것인가?**
   → 앱을 여는 것도 미룰 사람들이다. 알림·캘린더 연동·위젯이 없으면 **앱 자체가 미뤄진다.**
2. **뽀모도로 예측 정확도는 정말 사용자가 원하는 가치인가?**
   → 훌륭한 차별점이지만, 초보 사용자에겐 부담일 수 있음. **첫 화면엔 숨기고**, 데이터가
   쌓인 뒤 "당신은 자주 과소예측해요" 같은 *발견*으로 보여주는 게 안전.
3. **90% 산출물 기준을 사용자가 스스로 잘 쪼갤 수 있는가?**
   → 대부분 못 한다. **템플릿(보고서/기획서/과제용 90% 체크리스트 예시)**과 약한 AI 보조가
   없으면 "로켓 만들기"에서 막힌다. (브리프도 AI 자동분해를 후반으로 미뤄둠)

## 4. 추천 우선순위 (제품화 로드맵)

1. **먼저 진짜가 되게**: C-1 저장 · C-2 인증/RLS · C-3 타이머 정확도 · C-8 타임존
2. **공개 가능하게**: 2-1 개인정보방침/약관/탈퇴 · 2-2 보안 · 2-6 PWA 아이콘
3. **돌아오게**: 2-3 알림 루프 · 2-4 온보딩 · 2-5 분석/모니터링
4. **키우게**: P2(접근성·성능·운영·비용·랜딩)

> **한 줄 결론**: 지금은 "동작하는 데모"일 가능성이 큽니다. 제품이 되려면
> **① 데이터·인증이 진짜여야 하고 ② 타이머·타임존이 정확해야 하며 ③ 알림 루프와
> 개인정보 준비가 있어야** 합니다. 특히 이 앱은 *신뢰*와 *다시 오게 만드는 넛지*가
> 기능 목록보다 중요합니다.
